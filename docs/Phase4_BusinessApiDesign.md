# TouristSystem — Проектирование бизнес-модулей и API контрактов (Фаза 4)

Этот документ содержит подробный бизнес-анализ, сценарии использования (User Stories), спецификации DTO, правила валидации, матрицы прав доступа, спецификации эндпоинтов, бизнес-правила, обработку ошибок и стратегии тестирования для всех 9 основных модулей платформы **TouristSystem**.

---

## 1. Цель Фазы 4 (Phase 4 Goal)
Цель этой фазы — детально спроектировать функциональное поведение каждого модуля системы, зафиксировать контракты API и прописать бизнес-логику до написания кода. Это гарантирует, что при непосредственной разработке backend-а и frontend-а не возникнет разногласий по структуре данных и поведению системы.
В этой фазе **код backend-а и frontend-а не пишется**. Допускается только проектирование.

## 2. Проектирование модулей системы

### 2.1. Модуль авторизации и пользователей (Auth & Users Module)

#### 1. Purpose (Назначение)
Регистрация новых пользователей, безопасная аутентификация по протоколу JWT, ротация сессий через Refresh токен и базовая настройка профилей.

#### 2. User Stories (Пользовательские сценарии)
*   **Регистрация туриста:** Как гость, я хочу зарегистрироваться в системе с указанием ФИО, email и пароля, чтобы получить доступ к бронированию.
*   **Вход в систему:** Как пользователь, я хочу войти в систему по email и паролю, чтобы получить JWT Access Token.
*   **Ротация токенов:** Как вошедший пользователь, я хочу автоматически обновлять токен доступа с помощью токена обновления, чтобы сессия не прерывалась во время использования приложения.
*   **Выход (Logout):** Как пользователь, я хочу выйти из системы, чтобы мои токены авторизации были аннулированы на сервере в целях безопасности.

#### 3. Required DTOs (Необходимые DTO)
*   `RegisterRequestDto`: `[Required] string FullName`, `[Required] string Email`, `[Required] string Password`.
*   `LoginRequestDto`: `[Required] string Email`, `[Required] string Password`.
*   `TokenResponseDto`: `string AccessToken`, `DateTime AccessTokenExpiresAt`, `string RefreshToken`.
*   `ChangePasswordRequestDto`: `[Required] string OldPassword`, `[Required] string NewPassword`.

#### 4. Validation Rules (Правила валидации)
*   `Email`: Обязательно, формат электронной почты, максимальная длина 256 символов.
*   `Password`: Обязательно, длина не менее 8 символов, должен содержать минимум одну заглавную букву, одну цифру и один специальный символ.
*   `FullName`: Обязательно, длина от 2 до 100 символов, только буквы и пробелы.

#### 5. Permissions (Права доступа)
*   `Register`, `Login`, `RefreshToken`: Доступны анонимным пользователям (Anonymous).
*   `Logout`, `ChangePassword`, `GetMe`: Доступны только авторизованным пользователям (Authenticated).

#### 6. REST Endpoints (Эндпоинты)
*   `POST /api/auth/register` — Регистрация. Возвращает 201 Created.
*   `POST /api/auth/login` — Аутентификация. Записывает Refresh Token в HttpOnly secure cookie, возвращает Access Token в теле JSON.
*   `POST /api/auth/refresh-token` — Обновление токенов. Считывает Refresh Token из cookie, возвращает новые токены.
*   `POST /api/auth/logout` — Выход. Отзывает Refresh Token из базы данных, очищает cookie.
*   `PUT /api/users/change-password` — Смена пароля. Возвращает 200 OK.

#### 7. Business Rules (Бизнес-правила)
*   Пароли хранятся исключительно в виде хэша BCrypt с солью.
*   Refresh Token является одноразовым. При его использовании генерируется новая пара Access/Refresh токенов (механизм Refresh Token Rotation). Старый токен помечается как использованный (`is_revoked = true`).
*   После 5 неудачных попыток входа с одного email аккаунт временно блокируется на 15 минут.

#### 8. Edge Cases (Предельные случаи)
*   **Повторное использование Refresh Token (Token Reuse):** Если старый токен обновления, который уже был использован, отправляется повторно, сервер расценивает это как компрометацию сессии, аннулирует все активные токены обновления данного пользователя и требует повторного входа.

#### 9. Error Cases (Сценарии ошибок)
*   `400 Bad Request` — Ошибки валидации почты или пароля.
*   `401 Unauthorized` — Неверные учетные данные или просроченный токен сессии.
*   `409 Conflict` — Попытка зарегистрироваться на email, который уже занят.

#### 10. Testing Requirements (Требования к тестированию)
*   Unit-тесты для проверки валидатора паролей.
*   Тестирование генерации токенов JWT с использованием мока времени (`IDateTimeProvider`).
*   Интеграционный тест на ротацию токенов и сценарий компрометации (Token Reuse).

### 2.2. Модуль туристических мест (Places Module)

#### 1. Purpose (Назначение)
Ведение публичного каталога туристических мест и достопримечательностей.

#### 2. User Stories (Пользовательские сценарии)
*   **Просмотр каталога:** Как турист, я хочу видеть список мест с фильтром по городам и текстовым поиском, чтобы планировать путешествие.
*   **Предложение места:** Как турист или гид, я хочу предложить новую достопримечательность, заполнив форму с координатами и описанием.
*   **Модерация:** Как администратор, я хочу одобрять или отклонять предложенные места, чтобы поддерживать качество базы данных.

#### 3. Required DTOs (Необходимые DTO)
*   `CreatePlaceRequestDto`: `string Name`, `string Description`, `string City`, `string Address`, `decimal? Latitude`, `decimal? Longitude`, `decimal EntryFee`.
*   `PlaceResponseDto`: `Guid Id`, `string Name`, `string Slug`, `string Description`, `string City`, `string Address`, `decimal? Latitude`, `decimal? Longitude`, `string ImageUrl`, `decimal EntryFee`, `decimal RatingAverage`, `int ReviewsCount`, `string Status`.
*   `PlaceListDto`: Упрощенная модель для списков (без длинного описания).

#### 4. Validation Rules (Правила валидации)
*   `Name`: Обязательно, длина 3–150 символов.
*   `EntryFee`: Обязательно, значение должно быть больше или равно 0.
*   `Latitude`: От -90.000000 до 90.000000.
*   `Longitude`: От -180.000000 до 180.000000.

#### 5. Permissions (Права доступа)
*   Просмотр списка и деталей мест: Доступно анонимно (Anonymous).
*   Создание места (предложение): Любой авторизованный пользователь (Authenticated).
*   Редактирование, удаление, модерация: Только Администраторы (Admin, SuperAdmin).

#### 6. REST Endpoints (Эндпоинты)
*   `GET /api/places` — Поиск и пагинация мест.
*   `GET /api/places/{slug}` — Детали места по slug.
*   `POST /api/places` — Предложить место. Статус устанавливается в `Pending`.
*   `POST /api/places/{id}/moderate` — Изменить статус места (Approved/Rejected) (`Admin`).
*   `DELETE /api/places/{id}` — Мягкое удаление места (`Admin`).

#### 7. Business Rules (Бизнес-правила)
*   При создании места его `slug` автоматически генерируется из `Name` методом транслитерации и перевода в нижний регистр с дефисами (например, `озеро-байкал`).
*   Места в статусе `Pending` или `Rejected` скрыты для всех, кроме создателя места и администраторов.

#### 8. Edge Cases (Предельные случаи)
*   Создание места с именем, которое уже дает существующий `slug` -> к новому slug автоматически прибавляется случайный постфикс (например, `озеро-байкал-1`).

#### 9. Error Cases (Сценарии ошибок)
*   `404 Not Found` — Запрашиваемое место не найдено.
*   `403 Forbidden` — Попытка промодерировать место обычным пользователем.

#### 10. Testing Requirements (Требования к тестированию)
*   Тестирование корректности генерации `slug`.
*   Интеграционный тест на фильтрацию мест по названию города (регистронезависимый поиск).

### 2.3. Модуль отелей (Hotels Module)

#### 1. Purpose (Назначение)
Управление каталогом отелей, номерами и ценами для владельцев отелей.

#### 2. User Stories (Пользовательские сценарии)
*   **Регистрация отеля:** Как владелец отеля, я хочу добавить свой отель на платформу, чтобы привлекать туристов.
*   **Управление номерами:** Как владелец отеля, я хочу создавать типы номеров, указывать количество комнат и цену за ночь.
*   **Поиск отелей:** Как турист, я хочу искать отели в определенном городе, фильтруя их по цене, звездам и наличию Wi-Fi/парковки.

#### 3. Required DTOs (Необходимые DTO)
*   `CreateHotelRequestDto`: `string Name`, `string Description`, `string City`, `string Address`, `string PhoneNumber`, `decimal PricePerNight`, `int Stars`, `int TotalRooms`, `bool HasWifi`, `bool HasParking`, `bool HasPool`.
*   `HotelResponseDto`: Полная модель отеля, включая средний рейтинг и статус модерации.

#### 4. Validation Rules (Правила валидации)
*   `Stars`: Целое число от 1 до 5.
*   `PricePerNight`: Десятичное число строго больше 0.
*   `TotalRooms`: Больше 0.
*   `PhoneNumber`: Валидный международный формат телефона.

#### 5. Permissions (Права доступа)
*   Поиск и просмотр: Доступно анонимно.
*   Создание отеля: Доступно только пользователям с ролью `HotelOwner`.
*   Обновление информации, добавление комнат: Только создатель этого отеля (`HotelOwner` с проверкой `owner_id`).
*   Удаление и блокировка: Администраторы.

#### 6. REST Endpoints (Эндпоинты)
*   `GET /api/hotels` — Список отелей с фильтрами.
*   `GET /api/hotels/{id}` — Детали конкретного отеля.
*   `POST /api/hotels` — Регистрация отеля (`HotelOwner`).
*   `PUT /api/hotels/{id}` — Редактирование отеля (`HotelOwner` - владелец).
*   `DELETE /api/hotels/{id}` — Удаление отеля (Администратор/Владелец).

#### 7. Business Rules (Бизнес-правила)
*   Каждый отель должен пройти модерацию (статус `Pending` по умолчанию при создании). Только после одобрения администратором отель появляется в поиске для туристов.
*   Цены за ночь (`price_per_night`) должны быть валидными десятичными числами (тип `decimal` в C#).

#### 8. Edge Cases (Предельные случаи)
*   Владелец отеля пытается уменьшить количество комнат (`total_rooms`) ниже числа комнат, которые уже забронированы на будущие даты -> операция отклоняется с ошибкой валидации бизнес-правил.

#### 9. Error Cases (Сценарии ошибок)
*   `400 Bad Request` — Попытка установить отрицательную цену или некорректное число звезд.
*   `403 Forbidden` — Попытка отредактировать чужой отель.

#### 10. Testing Requirements (Требования к тестированию)
*   Unit-тесты бизнес-логики валидации владения отелем.
*   Интеграционные тесты поиска отелей по набору фильтров (цена + звезды + Wi-Fi).

### 2.4. Модуль ресторанов (Restaurants Module)

#### 1. Purpose (Назначение)
Предоставление туристам информации о ресторанах и кафе.

#### 2. User Stories (Пользовательские сценарии)
*   **Поиск ресторанов:** Как турист, я хочу фильтровать рестораны по типу кухни (например, грузинская, итальянская) и ценовому диапазону.
*   **Добавление ресторана:** Как владелец ресторана, я хочу внести свое заведение в систему, указав меню и часы работы.

#### 3. Required DTOs (Необходимые DTO)
*   `CreateRestaurantRequestDto`: `string Name`, `string Description`, `string City`, `string Address`, `string PhoneNumber`, `string CuisineType`, `string PriceRange` (Low/Medium/High), `string OpeningHours`.
*   `RestaurantResponseDto`: Модель ресторана с расчетом рейтинга.

#### 4. Validation Rules (Правила валидации)
*   `PriceRange`: Должен быть валидным значением перечисления (`Low`, `Medium`, `High`).
*   `OpeningHours`: Должно соответствовать паттерну времени (например, `HH:mm - HH:mm`).

#### 5. Permissions (Права доступа)
*   Поиск и просмотр: Доступно анонимно.
*   Создание ресторана: Только пользователи с ролью `RestaurantOwner`.
*   Редактирование: Владелец заведения.
*   Удаление: Администратор/Владелец.

#### 6. REST Endpoints (Эндпоинты)
*   `GET /api/restaurants` — Поиск и фильтрация ресторанов.
*   `POST /api/restaurants` — Регистрация ресторана (`RestaurantOwner`).
*   `PUT /api/restaurants/{id}` — Редактирование информации.

#### 7. Business Rules (Бизнес-правила)
*   Владелец может указать только один ценовой диапазон (`price_range`).
*   При добавлении отзывов пересчитывается средняя оценка ресторана.

#### 8. Edge Cases (Предельные случаи)
*   Ресторан работает в ночные часы (например, `18:00 - 04:00`). Валидатор должен корректно парсить диапазоны времени, переходящие через полночь.

#### 9. Error Cases (Сценарии ошибок)
*   `400 Bad Request` — Неверный формат времени работы.

#### 10. Testing Requirements (Требования к тестированию)
*   Тесты валидатора времени работы ресторана.
*   Интеграционные тесты фильтра по типу кухни.

### 2.5. Модуль транспорта (Transport Module)

#### 1. Purpose (Назначение)
Просмотр расписаний междугороднего транспорта и покупка билетов на свободные места.

#### 2. User Stories (Пользовательские сценарии)
*   **Поиск билетов:** Как турист, я хочу найти транспортные рейсы из города А в город Б на конкретную дату, чтобы спланировать переезд.
*   **Добавление рейса:** Как перевозчик, я хочу зарегистрировать рейс (автобус, минивэн), указав количество мест и цену билета.

#### 3. Required DTOs (Необходимые DTO)
*   `CreateTransportRequestDto`: `string Name`, `string Type`, `string OriginCity`, `string DestinationCity`, `DateTime DepartureTime`, `DateTime ArrivalTime`, `decimal PricePerSeat`, `int TotalSeats`, `string VehicleNumber`, `string ContactPhone`.
*   `TransportResponseDto`: Модель рейса с количеством оставшихся мест.

#### 4. Validation Rules (Правила валидации)
*   `DepartureTime`: Должна быть больше текущего времени (минимум на 2 часа вперед).
*   `ArrivalTime`: Должна быть строго больше `DepartureTime`.
*   `TotalSeats`: Минимум 1 место.
*   `PricePerSeat`: Больше или равно 0.

#### 5. Permissions (Права доступа)
*   Поиск рейсов: Анонимно.
*   Создание рейса: Только пользователи с ролью `TransportOwner`.
*   Редактирование/отмена рейса: Владелец рейса.

#### 6. REST Endpoints (Эндпоинты)
*   `GET /api/transports` — Поиск рейсов по параметрам (Origin, Destination, Date).
*   `POST /api/transports` — Создать рейс (`TransportOwner`).
*   `DELETE /api/transports/{id}` — Отменить рейс.

#### 7. Business Rules (Бизнес-правила)
*   При успешном бронировании билета туристом, поле `available_seats` в таблице `transports` уменьшается на количество купленных мест.
*   Если `available_seats` меньше запрашиваемого количества мест при покупке, транзакция отклоняется.

#### 8. Edge Cases (Предельные случаи)
*   **Параллельная покупка билетов (Race Condition):** Если два туриста одновременно пытаются купить последний билет, используется механизм оптимистичной блокировки (колонка `version` или токен конкурентности в EF Core). Один из запросов падает с ошибкой конкуренции, гарантируя, что овербукинга не произойдет.

#### 9. Error Cases (Сценарии ошибок)
*   `400 Bad Request` — Недостаточно свободных мест на рейсе.
*   `422 Unprocessable Entity` — Дата прибытия установлена раньше даты отправления.

#### 10. Testing Requirements (Требования к тестированию)
*   Стресс-тестирование конкурентной покупки билетов (Concurrency Integration Tests) на 10 параллельных потоках для проверки блокировок БД.

### 2.6. Модуль гидов (Guides Module)

#### 1. Purpose (Назначение)
Регистрация частных гидов, публикация их анкет и наем для экскурсий.

#### 2. User Stories (Пользовательские сценарии)
*   **Стать гидом:** Как пользователь, я хочу заполнить анкету гида (биография, языки, стоимость услуг в день), чтобы начать предлагать экскурсии.
*   **Поиск гида:** Как турист, я хочу найти гидов в городе N, говорящих на моем языке, чтобы забронировать их услуги.

#### 3. Required DTOs (Необходимые DTO)
*   `CreateGuideProfileDto`: `string Bio`, `string Languages` (список кодов через запятую, например "ru,en"), `string City`, `decimal PricePerDay`, `int ExperienceYears`.
*   `GuideProfileResponseDto`: Модель профиля со средним рейтингом и отзывами.

#### 4. Validation Rules (Правила валидации)
*   `Languages`: Обязательно, валидный формат кодов языков (ISO 639-1).
*   `ExperienceYears`: Целое число от 0 до 60.
*   `PricePerDay`: Десятичное число больше или равно 0.

#### 5. Permissions (Права доступа)
*   Просмотр анкет: Анонимно.
*   Создание профиля гида: Только пользователи с ролью `Guide`.
*   Редактирование своей анкеты: Только сам гид.

#### 6. REST Endpoints (Эндпоинты)
*   `GET /api/guides` — Поиск и фильтрация гидов (город, язык, цена).
*   `GET /api/guides/{id}` — Просмотр профиля гида.
*   `POST /api/guides/profile` — Создать профиль (`Guide`).
*   `PUT /api/guides/profile` — Обновить информацию профиля (`Guide`).

#### 7. Business Rules (Бизнес-правила)
*   Один пользователь может иметь только один профиль гида.
*   Новые профили гидов по умолчанию имеют статус `Pending` и скрыты из поиска до прохождения модерации администратором (проверка документов).

#### 8. Edge Cases (Предельные случаи)
*   Гид меняет город работы -> все активные бронирования гида в старом городе остаются в силе, но новые заявки принимаются только для нового города.

#### 9. Error Cases (Сценарии ошибок)
*   `409 Conflict` — Пользователь уже имеет профиль гида.

#### 10. Testing Requirements (Требования к тестированию)
*   Тестирование парсинга строки языков и фильтрации по ней (например, поиск гидов с языком "en").

### 2.7. Модуль бронирований (Bookings Module)

#### 1. Purpose (Назначение)
Ядро системы: оформление заказов услуг, расчет стоимости, контроль жизненного цикла бронирования.

#### 2. User Stories (Пользовательские сценарии)
*   **Бронирование:** Как турист, я хочу выбрать отель/рейс/гида, указать даты и количество человек, чтобы создать бронирование.
*   **Управление бронью:** Как владелец бизнеса, я хочу подтверждать или отклонять входящие заявки на бронирование моих услуг.
*   **Отмена брони:** Как турист, я хочу отменить свое бронирование, если мои планы изменились.

#### 3. Required DTOs (Необходимые DTO)
*   `CreateBookingRequestDto`: `[Required] string BookingType` (Hotel/Transport/Guide), `[Required] Guid ReferenceId`, `[Required] DateTime StartDate`, `[Required] DateTime EndDate`, `[Required] int GuestsCount`, `int Quantity`.
*   `BookingResponseDto`: Полная информация о заказе, включая статус (`Pending`, `Confirmed`, `Cancelled`, `Expired`, `Completed`) и `TotalPrice`.

#### 4. Validation Rules (Правила валидации)
*   `StartDate`: Должна быть больше или равна сегодняшней дате (UTC).
*   `EndDate`: Должна быть больше или равна `StartDate`.
*   `GuestsCount`: Целое число строго больше 0.
*   `Quantity`: Больше 0 (количество бронируемых комнат или мест).

#### 5. Permissions (Права доступа)
*   Создание, просмотр своих бронирований: `Tourist` (Authenticated).
*   Подтверждение, отклонение: Владелец услуги (`Guide`, `HotelOwner`, `TransportOwner`, `RestaurantOwner`).
*   Просмотр всех бронирований в системе: Администраторы (`Admin`, `SuperAdmin`).

#### 6. REST Endpoints (Эндпоинты)
*   `POST /api/bookings` — Создать бронирование. Возвращает 201 Created и статус `Pending`.
*   `GET /api/bookings/my` — История бронирований туриста.
*   `POST /api/bookings/{id}/confirm` — Подтвердить бронирование (доступно владельцу услуги).
*   `POST /api/bookings/{id}/cancel` — Отменить бронирование (доступно туристу или владельцу).

#### 7. Business Rules (Бизнес-правила)
*   При создании бронирования цена фиксируется в поле `TotalPrice` как снимок цены на дату заказа и больше не меняется при изменении тарифов поставщика услуг.
*   **Стейт-машина бронирования:**
    ```
    [Pending] ──(Подтверждение владельцем)──► [Confirmed] ──(Срок прошел)──► [Completed]
        │                                         │
    (Отмена / Неоплата)                       (Отмена)
        │                                         │
        ▼                                         ▼
    [Expired / Cancelled]                    [Cancelled]
    ```
*   Если статус `Pending` не подтвержден владельцем в течение 24 часов, фоновая служба сервера автоматически переводит бронирование в статус `Expired` и освобождает заблокированные места/номера.

#### 8. Edge Cases (Предельные случаи)
*   Бронирование гида на накладывающиеся даты -> система перед созданием проверяет календарь занятости гида, если даты пересекаются с другим подтвержденным бронированием, выводится ошибка.

#### 9. Error Cases (Сценарии ошибок)
*   `400 Bad Request` — Даты бронирования некорректны или пересекаются с существующими заказами.
*   `422 Unprocessable Entity` — Объект бронирования заблокирован или удален.

#### 10. Testing Requirements (Требования к тестированию)
*   Unit-тесты для стейт-машины переходов статусов бронирования.
*   Интеграционные тесты фоновой службы отмены просроченных броней (проверка перехода `Pending -> Expired` через 24 часа).

### 2.8. Модуль отзывов (Reviews Module)

#### 1. Purpose (Назначение)
Сбор обратной связи от туристов, расчет рейтингов для отелей, ресторанов, мест и гидов.

#### 2. User Stories (Пользовательские сценарии)
*   **Оставить отзыв:** Как турист, я хочу написать отзыв о посещенном отеле/гиде и выставить оценку от 1 до 5 звезд, чтобы поделиться опытом с другими.
*   **Модерация отзывов:** Как администратор, я хочу удалять отзывы, содержащие спам или нецензурную лексику.

#### 3. Required DTOs (Необходимые DTO)
*   `CreateReviewRequestDto`: `[Required] string ReviewType` (Place/Hotel/Guide/Restaurant), `[Required] Guid ReferenceId`, `[Required] int Rating`, `[Required] string Comment`.
*   `ReviewResponseDto`: Данные отзыва, включая имя автора и дату публикации.

#### 4. Validation Rules (Правила валидации)
*   `Rating`: Целое число от 1 до 5.
*   `Comment`: Обязательно, длина от 10 до 1000 символов.

#### 5. Permissions (Права доступа)
*   Чтение отзывов: Анонимно.
*   Написание отзыва: Авторизованный пользователь.
*   Удаление отзыва: Только Администраторы.

#### 6. REST Endpoints (Эндпоинты)
*   `POST /api/reviews` — Оставить отзыв.
*   `GET /api/reviews/{type}/{referenceId}` — Список отзывов о конкретном объекте.
*   `DELETE /api/reviews/{id}` — Удалить отзыв (`Admin`).

#### 7. Business Rules (Бизнес-правила)
*   Пользователь может оставить **только один отзыв** на один конкретный объект. Повторные отзывы перезаписывают старый отзыв или отклоняются.
*   При добавлении, обновлении или удалении отзыва система автоматически пересчитывает поля `RatingAverage` и `ReviewsCount` у целевого объекта (отеля, места, ресторана, гида) в единой базе данных.

#### 8. Edge Cases (Предельные случаи)
*   Удаление последнего отзыва у отеля -> средний рейтинг `RatingAverage` сбрасывается в `0.00`, а `ReviewsCount` в `0`.

#### 9. Error Cases (Сценарии ошибок)
*   `400 Bad Request` — Попытка отправить отзыв с оценкой 6 звезд или дублирующий отзыв.

#### 10. Testing Requirements (Требования к тестированию)
*   Unit-тест формулы перерасчета среднего рейтинга при добавлении нового отзыва.

### 2.9. Модуль администратора (Admin Module)

#### 1. Purpose (Назначение)
Панель управления платформой, модерация заявок бизнеса, просмотр логов безопасности и аналитики.

#### 2. User Stories (Пользовательские сценарии)
*   **Просмотр дашборда:** Как администратор, я хочу видеть общую статистику (количество пользователей, отелей, бронирований, выручку), чтобы оценивать работу платформы.
*   **Управление статусом пользователей:** Как администратор, я хочу блокировать нарушителей правил системы.
*   **Анализ логов аудита:** Как супер-администратор, я хочу просматривать историю изменений критических данных, чтобы расследовать инциденты безопасности.

#### 3. Required DTOs (Необходимые DTO)
*   `AdminDashboardStatsDto`: Агрегированные данные статистики (счетчики, списки последних событий).
*   `AuditLogResponseDto`: Модель лога аудита с возможностью чтения старых и новых значений (`old_values`, `new_values`).
*   `UserStatusUpdateRequestDto`: `bool IsActive`, `string BlockReason`.

#### 4. Validation Rules (Правила валидации)
*   `BlockReason`: Обязательно при блокировке пользователя (`IsActive == false`), минимальная длина 10 символов.

#### 5. Permissions (Права доступа)
*   Модерация контента, просмотр статистики, блокировка пользователей: Роли `Admin`, `SuperAdmin`.
*   Управление списком администраторов, чтение логов аудита (`AuditLogs`): Только `SuperAdmin`.

#### 6. REST Endpoints (Эндпоинты)
*   `GET /api/admin/stats` — Статистика для графиков дашборда.
*   `GET /api/admin/audit-logs` — История изменений системы (только `SuperAdmin`).
*   `PUT /api/admin/users/{id}/status` — Заблокировать/разблокировать пользователя.

#### 7. Business Rules (Бизнес-правила)
*   Каждое изменение данных через эндпоинты администратора должно создавать новую запись в таблице `audit_logs` с фиксацией IP адреса и User-Agent.
*   Администратор не может заблокировать или изменить статус роли `SuperAdmin`.

#### 8. Edge Cases (Предельные случаи)
*   Логирование больших объектов в `old_values`/`new_values` -> сериализатор JSON игнорирует бинарные данные (например, загруженные картинки отелей) во избежание переполнения БД.

#### 9. Error Cases (Сценарии ошибок)
*   `403 Forbidden` — Попытка обычного администратора получить доступ к логам аудита или управлению ролями.

#### 10. Testing Requirements (Требования к тестированию)
*   Тестирование перехвата изменений и автоматической записи логов аудита при редактировании сущности пользователя.
