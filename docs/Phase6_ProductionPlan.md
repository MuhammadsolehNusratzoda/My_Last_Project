# TouristSystem — DevOps, Безопасность, Тестирование и Production (Фаза 6)

Этот документ содержит полное описание стратегий тестирования, безопасности, производительности, мониторинга, резервного копирования и развертывания платформы **TouristSystem** в промышленную среду (Production) с помощью контейнеризации Docker.

---

## 1. Цель Фазы 6 (Phase 6 Goal)
Цель этой фазы — подготовить проект к enterprise-развертыванию, минимизировать риски безопасности (OWASP), повысить производительность системы под нагрузкой, спроектировать Docker-инфраструктуру и настроить логирование и резервное копирование.
В этой фазе **код инфраструктуры (Dockerfiles, compose.yaml, конфиги Nginx, CI/CD скрипты) не пишется**. Допускается только проектирование.

---

## 2. Разница между Development и Production
В среде разработки (Development) мы отдаем приоритет скорости написания кода, отладке и простоте запуска (все сервисы в `localhost`, подробные логи в консоль, отключенные лимиты). 
В промышленной среде (Production) приоритетами становятся:
*   **Безопасность (Security):** Ограничение прав доступа, шифрование трафика (SSL), защита от атак.
*   **Высокая доступность (High Availability) и отказоустойчивость:** Если упадет один контейнер, система должна автоматически перезапустить его без прерывания работы.
*   **Изоляция данных:** База данных должна быть закрыта от прямого доступа извне.
*   **Производительность:** Сжатие ресурсов, кэширование, минимизация размера образов.

## 3. Стратегия тестирования (Testing Plan)

Мы внедряем пирамиду тестирования для контроля качества на всех уровнях.

```
       / \       E2E Tests (Playwright / Cypress) - Мало, дорого
      /   \      API Tests (Postman / RestSharp) - Средне
     /     \     Integration Tests (xUnit + WebApplicationFactory) - Средне
    /_______\    Unit Tests (xUnit + Mock + FluentAssertions) - Много, быстро
```

### 3.1. Тестирование Backend (C# / .NET 9)
*   **Unit Tests (Модульные тесты):**
    *   *Что тестируем:* Валидаторы (`FluentValidation`), маппинги, вспомогательные утилиты, доменную логику сущностей (например, переходы статусов бронирования).
    *   *Инструменты:* `xUnit`, `Moq` (для изоляции зависимостей), `FluentAssertions` (для читаемых проверок).
*   **Integration Tests (Интеграционные тесты):**
    *   *Что тестируем:* Бизнес-сценарии в Application слое с реальным подключением к тестовой БД PostgreSQL (используем `Testcontainers` для запуска PostgreSQL в Docker во время тестов). Проверяем корректность работы репозиториев и коммит транзакций.
*   **API Tests (Тесты эндпоинтов):**
    *   *Что тестируем:* Заголовки ответов, коды статусов (200, 201, 400, 401, 403), обработку middleware. Используется `Microsoft.AspNetCore.Mvc.Testing` (`WebApplicationFactory`).

### 3.2. Тестирование Frontend (React / TypeScript)
*   **Component Tests (Тестирование компонентов):**
    *   *Что тестируем:* Корректность рендеринга общих компонентов (кнопки, формы ввода, карточки), состояние лоадеров при загрузке.
    *   *Инструменты:* `Vitest`, `React Testing Library`.
*   **E2E Tests (Сквозные тесты):**
    *   *Что тестируем:* Полные сценарии пользователя: «Регистрация -> Авторизация -> Поиск отеля -> Бронирование -> Проверка истории бронирований».
    *   *Инструменты:* `Playwright` или `Cypress`.

## 4. Чек-лист безопасности (Security Checklist)

Мы следуем стандартам OWASP для обеспечения промышленного уровня безопасности:

1.  **Защита JWT и Refresh-токенов:**
    *   Access Token имеет короткий срок жизни (15 минут), передается в заголовке `Authorization: Bearer`.
    *   Refresh Token записывается в зашифрованные `HttpOnly`, `Secure`, `SameSite=Strict` cookies. Это полностью предотвращает доступ к нему через JS (защита от XSS-уязвимостей).
2.  **Шифрование трафика (HTTPS):**
    *   Весь трафик между клиентом, прокси-сервером и API шифруется через TLS 1.3. Сертификаты генерируются автоматически через `Let's Encrypt`.
3.  **Политика CORS (Cross-Origin Resource Sharing):**
    *   Backend разрешает запросы только от доверенного домена фронтенда (например, `https://touristsystem.com`). Запросы от сторонних доменов блокируются на уровне WebApi.
4.  **Rate Limiting (Ограничение частоты запросов):**
    *   Защита от DDoS и брутфорса. Внедряется ограничение частоты запросов (например, максимум 100 запросов в минуту с одного IP, для эндпоинта логина — не более 5 в минуту). Настраивается в Nginx или через middleware в ASP.NET Core.
5.  **Защита от SQL-инъекций:**
    *   Использование `Entity Framework Core` исключает SQL-инъекции, так как все SQL-запросы параметризованы по умолчанию. Прямые SQL-запросы (`FromSqlRaw`) запрещены без строгого код-ревью.
6.  **Защита от XSS (Cross-Site Scripting):**
    *   Фронтенд на React автоматически экранирует выводимый текст.
    *   Используются заголовки безопасности HTTP (`Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`).
7.  **Secret Management (Управление секретами):**
    *   Все секреты (строка подключения к БД, JWT-ключ, SMTP-пароли) выносятся из кода. В Development они хранятся в `User Secrets` (или локальном `.env`), в Production передаются как переменные окружения (Environment Variables) в Docker-контейнеры.
8.  **Защита прав доступа (BOLA/IDOR):**
    *   При обновлении/удалении отеля/ресторана/бронирования проверяется не только авторизация, но и принадлежность ресурса пользователю (`resource.OwnerId == currentUserId`).

## 5. Производительность и оптимизация (Performance Checklist)

1.  **Индексация базы данных:** Индексы на полях поиска/фильтрации (см. Раздел 6 Фазы 2).
2.  **Пагинация по умолчанию:** Все эндпоинты, отдающие списки (места, отели, бронирования), возвращают данные страницами (`pageNumber` и `pageSize`), предотвращая перегрузку памяти.
3.  **Кэширование (Caching):**
    *   Внедрение кэширования для редко меняющихся данных (список городов, детали достопримечательностей) с помощью `InMemoryCache` или распределенного кэша `Redis`.
    *   Использование заголовков HTTP-кэширования (`ETag`, `Cache-Control`) на стороне Nginx.
4.  **Сжатие ответов (Compression):**
    *   Включение сжатия `Gzip` или `Brotli` для JSON-ответов API и статических файлов фронтенда.
5.  **Оптимизация изображений:**
    *   Все загружаемые пользователями изображения сжимаются на бэкенде перед сохранением в WebP-формат (разрешение до Full HD).
6.  **Lazy Loading (Ленивая загрузка):**
    *   На фронтенде страницы загружаются динамически (`React.lazy` + `Suspense`), разбивая общий JS-бандл на части (Code Splitting).

## 6. Схема развертывания (Deployment Plan)

Для изоляции и простоты масштабирования развертывание выполняется в контейнерах **Docker** с помощью оркестратора **Docker Compose**.

```
                           [ Клиент / Браузер ]
                                    │
                                    ▼ (Порты 80 / 443 - HTTPS)
                        [ Nginx Reverse Proxy Container ]
                                    │
            ┌───────────────────────┴───────────────────────┐
            ▼ (Статические файлы)                           ▼ (Проксирование /api)
[ Frontend React Container ]                     [ Backend WebApi Container ]
                                                            │
                                                            ▼ (Внутренняя сеть Docker)
                                                 [ PostgreSQL Database Container ]
```

### Описание контейнеров:
1.  **Nginx (Reverse Proxy):**
    *   Точка входа. Принимает HTTPS-запросы, терминирует SSL-сертификаты.
    *   Раздает собранные статические файлы фронтенда напрямую.
    *   Запросы, начинающиеся на `/api`, перенаправляет (проксирует) на контейнер с ASP.NET Core Web API.
2.  **Frontend (React SPA):**
    *   Контейнеризируется на базе многоэтапной сборки (Multi-stage build). Сначала Node.js собирает продакшн-версию приложения, затем собранные HTML/JS/CSS копируются в легковесный образ Nginx.
3.  **Backend (ASP.NET Core .NET 9):**
    *   Запускается в режиме `Release`. Для уменьшения размера используется базовый образ `.NET Runtime` (около 200 МБ) вместо полного образа SDK.
4.  **PostgreSQL (Database):**
    *   Официальный образ PostgreSQL. Данные монтируются в Docker Volume на хост-машину (`pgdata`), что предотвращает их удаление при перезапуске контейнера.
    *   Доступ к порту 5432 закрыт извне. К БД может подключиться только контейнер бэкенда по внутренней Docker-сети.

## 7. Мониторинг и логирование (Monitoring Strategy)

1.  **Логирование (Serilog):**
    *   Логи пишутся в структурированном JSON-формате.
    *   В Development логи выводятся в консоль. В Production логи записываются в файлы с ротацией (каждый день создается новый файл, старые логи сжимаются и удаляются через 14 дней) и отправляются в систему сбора логов (например, Elasticsearch / Kibana или Seq).
2.  **Уровни логирования:**
    *   `Information` — Запуск приложения, миграции, успешные важные бизнес-операции.
    *   `Warning` — Ошибки валидации запросов, попытки несанкционированного доступа.
    *   `Error` — Системные исключения, падение соединений с БД.
3.  **Health Checks:**
    *   Служба Docker перезапускает контейнеры бэкенда (`restart: always`), если системный эндпоинт `/health` перестает отвечать.

## 8. Резервное копирование и Disaster Recovery (Backup & Rollbacks)

*   **Резервное копирование БД:**
    *   Скрипт по крону (`cron`) раз в сутки делает резервную копию базы данных с помощью утилиты `pg_dump`.
    *   Дампы сжимаются в `.tar.gz`, шифруются и загружаются в удаленное облачное хранилище (хостинг дампов). Хранится история дампов за последние 30 дней.
*   **Стратегия отката (Rollback):**
    *   Каждый Docker-образ тегируется версией (например, `v1.0.1`, `v1.0.2`), привязанной к коммиту в GitHub. Тег `latest` в продакшене не используется.
    *   В случае сбоя новой версии развертывания администратор выполняет команду отката на предыдущий тег образа за 1 минуту:
        ```bash
        docker compose up -d --build
        ``` (с изменением версии в конфигурационном файле).

## 10. Долгосрочное развитие системы (Long-Term Evolution)

Для обеспечения возможности роста TouristSystem до миллионов пользователей архитектура поддерживает:
1.  **Stateless Backend:** Сервер бэкенда не хранит сессионных файлов в своей памяти. Все сессии авторизации хранятся в токенах и БД. Это позволяет в будущем запустить 5-10 параллельных контейнеров бэкенда за балансировщиком нагрузки (Nginx/HAProxy).
2.  **Интеграция платежей:** Таблица `bookings.total_price` содержит снимок цены, готовый для передачи в Stripe/PayPal API в последующих фазах.
3.  **Поддержка CDN:** Раздача тяжелых файлов картинок мест и отелей через сеть серверов CDN (Cloudflare) для снижения трафика на наш сервер.
