# TouristSystem — Проектирование React Фронтенда (Фаза 5)

Этот документ детально описывает клиентскую архитектуру, структуру папок, разметку страниц, схему маршрутизации, архитектуру управления состоянием (Zustand), интеграцию с API и правила UX/UI дизайна для фронтенд-приложения **TouristSystem** на React 19 и TypeScript.

---

## 1. Цель Фазы 5 (Phase 5 Goal)
Цель этой фазы — спроектировать всю структуру фронтенд-приложения, определить контракты обмена данными с API на клиенте, прописать логику роутинга и защищенных маршрутов, детально распланировать переиспользуемые UI-компоненты и логику хранения сессии.
В этой фазе **код компонентов React, хуков, стилей CSS не пишется**. Допускается только проектирование.

---

## 2. Архитектура Frontend-приложения (Frontend Architecture)

Клиентское приложение представляет собой Single Page Application (SPA), собранное на базе **Vite** с использованием **React 19** и строгого **TypeScript**. 

### Стек технологий:
*   **Сборщик:** Vite (быстрая сборка, Hot Module Replacement).
*   **Маршрутизация:** React Router (клиентский роутинг, вложенные маршруты).
*   **Состояние:** Zustand (легковесный глобальный стейт-менеджер для сессий и настроек).
*   **Запросы к API:** Axios (обертка для HTTP-запросов) + TanStack Query (кэширование данных, обработка статусов Loading/Error/Success).
*   **Стили и UI:** Tailwind CSS (утилитарные стили) + Shadcn UI (доступные компоненты на базе Radix UI).
*   **Формы и валидация:** React Hook Form + Zod (типизированная валидация схем на клиенте).
*   **Анимации:** Framer Motion (плавные переходы страниц и микро-анимации интерфейса).

## 3. Структура каталогов (Folder Structure)

Мы используем гибридную структуру: общие ресурсы сгруппированы по техническому назначению, а бизнес-логика разделена на функциональные модули (Features).

```
src/
├── app/                  # Инициализация приложения, глобальные провайдеры
│   ├── api.ts            # Конфигурация Axios и интерцепторов
│   └── store.ts          # Zustand stores (authStore, uiStore)
├── assets/               # Статические ресурсы (логотипы, шрифты, иконки)
├── components/           # Общие атомарные UI компоненты (Button, Input, Modal)
├── features/             # Бизнес-модули (инкапсулируют логику фичи)
│   ├── auth/             # Компоненты Login/Register, хуки авторизации
│   ├── hotels/           # Карточка отеля, список отелей, формы бронирования отеля
│   └── places/           # Карточка места, сетка достопримечательностей
├── hooks/                # Переиспользуемые хуки (useAuth, useDebounce, useTheme)
├── layouts/              # Макеты страниц (Header/Sidebar/Footer wrappers)
├── pages/                # Страницы-контейнеры (связывают UI с роутером)
├── routes/               # Конфигурация путей и защищенные маршруты (Route Guards)
├── services/             # API клиенты (placesService, hotelService)
├── types/                # Глобальные файлы описания типов TypeScript
└── utils/                # Хелперы форматирования дат, строк, цен
```

## 4. Страницы приложения (Pages Breakdown)

Система содержит 21 страницу, разделенную по уровням доступа.

### 4.1. Публичные страницы (Public Pages)
1.  **Home (Главная):** Промо-баннер с умным поиском по городам и категориям, сетка популярных достопримечательностей.
2.  **Login (Вход):** Форма ввода email/пароля с обработкой ошибок авторизации.
3.  **Register (Регистрация):** Форма создания аккаунта туриста.
4.  **Places (Достопримечательности):** Каталог мест с фильтрами по городу, цене входа, текстовым поиском и пагинацией.
5.  **Place Details (Детали места):** Описание, фотогалерея, координаты на карте, отзывы пользователей.
6.  **Hotels (Отели):** Поиск отелей по городам, датам заезда/выезда, количеству звезд, ценовому диапазону.
7.  **Hotel Details (Детали отеля):** Описание отеля, список доступных номеров, удобства, отзывы туристов.
8.  **Restaurants (Рестораны):** Список заведений с фильтром по типу кухни и среднему чеку.
9.  **Transport (Транспорт):** Поиск билетов на автобусы, поезда, трансферы (Откуда, Куда, Дата).
10. **Guides (Гиды):** Каталог гидов с ценами за день, языками и рейтингом.

### 4.2. Приватные страницы туриста (Authenticated Pages)
11. **Booking Create (Создание бронирования):** Интерактивная форма выбора дат, количества гостей и расчета итоговой цены перед покупкой.
12. **My Bookings (Мои бронирования):** Личный кабинет со списком прошлых и активных заказов, статусами оплаты, кнопкой отмены.
13. **Profile (Профиль):** Управление личными данными (ФИО, аватар, телефон), смена пароля.
14. **Favorites (Избранное):** Список сохраненных мест, отелей и ресторанов.

### 4.3. Страницы владельцев бизнеса (Owner Dashboards)
15. **Owner Dashboard (Панель отельера/перевозчика/ресторатора):** Календарь заездов, входящие заявки на бронирование, статистика просмотров.
16. **My Services Management:** Форма создания/редактирования отеля, добавления номеров, расписания рейсов.

### 4.4. Панель Администратора (Admin Dashboards)
17. **Admin Dashboard (Панель админа):** Графики популярности, статистика верификации гидов, аудит.
18. **Admin Users:** Управление пользователями, блокировка нарушителей.
19. **Admin Places:** Модерация предложенных пользователями достопримечательностей.
20. **Admin Bookings / Reviews:** Контроль конфликтных бронирований, удаление недопустимых отзывов.
21. **Audit Logs Page (SuperAdmin):** Просмотр детальных системных журналов аудита.

## 5. UI Компоненты и API Пропсов (UI Components Design)

Каждый компонент строго типизирован. Ниже приведен дизайн ключевых компонентов:

1.  **Navbar (Шапка сайта):**
    *   *Пропсы:* Нет. Получает состояние `isLoggedIn`, `userRole` из Zustand `authStore`.
    *   *Поведение:* Меняет меню в зависимости от роли (турист видит Избранное, Владелец — Панель Услуг, Админ — Панель Модерации). Поддерживает переключение темной темы.
2.  **Card (Карточка объекта):**
    *   *Пропсы:* `id UUID`, `title string`, `imageUrl string`, `rating number`, `price number`, `location string`, `type "hotel" | "place" | "restaurant" | "guide"`.
    *   *Поведение:* Клик ведет на страницу деталей. Содержит иконку «Добавить в избранное» с отправкой запроса к API.
3.  **BookingForm (Форма бронирования):**
    *   *Пропсы:* `targetId UUID`, `bookingType string`, `pricePerUnit number`, `onSuccess function`.
    *   *Поведение:* Использует `React Hook Form` для ввода дат. Валидирует, что дата начала не в прошлом. Динамически рассчитывает `total_price = days * pricePerUnit` и выводит пользователю.
4.  **SearchBar (Строка поиска):**
    *   *Пропсы:* `placeholder string`, `onSearchChange (query: string) => void`.
    *   *Поведение:* Реализует debounce (задержку 300мс) перед отправкой поискового запроса на сервер во избежание лишней нагрузки на API.
5.  **Modal (Модальное окно):**
    *   *Пропсы:* `isOpen boolean`, `title string`, `onClose function`, `children ReactNode`.
    *   *Поведение:* Использует Radix UI Dialog для фокуса, закрывается по ESC и клику на оверлей.

## 6. Управление состоянием (State Management)

Мы используем **Zustand** для управления легкими глобальными состояниями. Данные о результатах запросов к API кэшируются через **TanStack Query**.

### Zustand stores:
1.  **AuthStore (Сессия):**
    *   `accessToken`: строка в памяти (не сохраняется в localStorage для защиты от XSS).
    *   `user`: объект текущего пользователя (ID, имя, email, роль).
    *   `login(token, user)`: метод инициализации сессии.
    *   `logout()`: очистка сессии.
2.  **UIStore (Интерфейс):**
    *   `theme`: `"light" | "dark"`. Автоматически синхронизируется с `localStorage` и добавляет класс `.dark` на элемент `<html>`.
    *   `sidebarOpen`: состояние открытия меню на мобильных устройствах.

## 7. Интеграция с API и ротация токенов (API & Security Flow)

Вся коммуникация с C# Web API происходит через настроенный инстанс **Axios**.

```
   React Client                    Backend API
        │                               │
        │ ─── 1. Send Request ────────► │
        │     (Bearer AccessToken)      │
        │                               │
        │ ◄── 2. Error 401 Unauth ───── │ (AccessToken expired)
        │                               │
        │ ─── 3. POST /refresh-token ──►│ (Automatic Interceptor call)
        │     (HttpOnly Refresh Cookie) │
        │                               │
        │ ◄── 4. New AccessToken ────── │ (Tokens Rotated successfully)
        │                               │
        │ ─── 5. Retry Original Req ──► │
        │     (Bearer NewAccessToken)   │
        │                               │
        │ ◄── 6. Response 200 OK ────── │
```

### Axios Interceptors (Перехватчики):
1.  **Request Interceptor:** Перед отправкой любого запроса считывает `accessToken` из Zustand store и подставляет его в заголовок `Authorization: Bearer <token>`.
2.  **Response Interceptor:** Если сервер вернул `401 Unauthorized`:
    *   Запрос ставится "на паузу".
    *   Axios делает автоматический фоновый запрос `POST /api/auth/refresh-token`.
    *   Если токен обновлен успешно, мы сохраняем новый `AccessToken` в Zustand и повторяем все приостановленные запросы с новым токеном.
    *   Если запрос обновления упал с ошибкой (Refresh Token просрочен/отозван), пользователя автоматически перенаправляет на страницу `/login` с выводом сообщения «Сессия истекла».

## 8. Дизайн-система и UX Правила (Design System & UX Rules)

### Цветовая палитра и Typography:
*   **Стиль:** Современный легкий дизайн (Modern Tourism Style). Использование скругленных углов (`rounded-2xl`), эффекта стекла (Glassmorphism для шапки), мягких теней.
*   **Цвета (Tailwind):**
    *   *Light Mode:* Фоновый `#F8FAFC` (slate-50), основной текст `#0F172A` (slate-900), акцентный `#0EA5E9` (sky-500).
    *   *Dark Mode:* Фоновый `#0B0F19` (slate-950), текст `#F1F5F9` (slate-100), акцентный `#38BDF8` (sky-400).
*   **Шрифты:** Семейство шрифтов **Inter** или **Outfit** от Google Fonts.

### UX Правила загрузки данных:
*   **Skeleton Loaders:** Вместо серых индикаторов загрузки (Spinners) при загрузке каталогов отелей или мест выводятся скелетоны (скелетные карточки) для повышения воспринимаемой скорости работы приложения.
*   **Debounce поиска:** Поисковые инпуты реагируют на ввод с задержкой в 300мс (lodash.debounce или хук `useDebounce`), отправляя запрос только тогда, когда пользователь закончил вводить текст.
*   **Toast Notifications:** Все критические действия (успешное бронирование, ошибка авторизации) сопровождаются всплывающими плашками Toasts в углу экрана (Shadcn toast).
